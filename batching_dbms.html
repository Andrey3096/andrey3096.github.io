Batch Deletes and Updates in PostgreSQL

Why large single-transaction changes are risky, when they are acceptable, and how to process data in smaller, safer batches.

Problem

Why a single large transaction can hurt

Locks Autovacuum and bloat No progress persistence WAL growth and replication pressure Uncontrolled I/O load

Locks

At a high level, there are two kinds of locks involved: table-level and row-level.

Whenever you modify rows, PostgreSQL takes a Row Exclusive lock on the table. It is not the strictest one, but it still conflicts with operations like creating indexes, adding triggers, running REFRESH MATERIALIZED VIEW CONCURRENTLY, or altering the table structure. If one of those operations kicks in during a long-running delete, it will simply have to wait until your transaction finishes — which can lead to long and unpredictable delays.

On the row level, things are even more straightforward. PostgreSQL does not allow two transactions to modify the same row version at the same time. So if another transaction tries to update or delete the same rows, it just waits in line. If your transaction is long-running, you can easily end up blocking other queries for a long time.

Batching keeps transactions short.

Autovacuum and bloat

A long-running single-transaction UPDATE or DELETE holds back the database horizon (xmin).

As a result, autovacuum cannot remove dead tuples that may still be visible to an active transaction. If this situation persists, table and index bloat can accumulate over time.

Batching helps avoid holding back the database horizon.

No progress persistence

A long-running transaction does not save intermediate progress. If it fails or gets interrupted, everything is rolled back and you have to start from scratch — which is especially painful when dealing with millions of rows.

Batching preserves committed progress.

WAL growth and replication pressure

Every UPDATE or DELETE generates WAL. When a huge amount of data is changed in a single transaction, PostgreSQL has to produce a large burst of WAL in a short period of time. That burst increases disk pressure on the primary and can also create a sudden backlog for replicas.

Batching makes the flow of changes more predictable. By adding pauses between batches, you can spread the workload over time, slow down WAL growth, and give replicas a realistic chance to consume the stream of changes without falling badly behind.

Batching makes the flow of changes more predictable.

Uncontrolled I/O load

Changing rows also changes data pages, which makes shared buffers dirty. Later, those dirty buffers have to be written to disk by background processes or by user backends. A massive update or delete can dirty a large part of the buffer cache at once, and if the storage is already busy, flushing all those pages may become expensive for the system.

With batched updates or deletes, you can regulate I/O pressure by adding a delay between batches. This spreads the work over time and makes writes smoother instead of turning them into one large spike.

Batching keeps I/O load under control.

Exceptions

When a single transaction can be acceptable

Small data volume

What counts as “small” depends on your workload and SLOs.

For high-load systems, even a batch of 10k rows (actual number depends on a row size in bytes) may already be near the safe limit — increasing it further can lead to lock contention and service degradation.

On the other hand, for mixed workloads or OLAP systems, deleting a few million rows in a single transaction can be perfectly acceptable.

Deleting historical data

If the data is no longer used by the application, it is generally safe to delete it in a single operation.

This can involve millions of rows. However, if the volume is larger, batching is still recommended to preserve progress and reduce operational risk.

PostgreSQL examples

Index exists

Batching by time interval

This approach works well when the cleanup condition is based on an indexed timestamp column. In this case, we can delete old rows in small time windows instead of removing everything in one large transaction.

DO
$$
DECLARE
    batch_interval interval := '1 hour';
    end_timestamp  timestamptz := now() - interval '10 days';
    left_ts        timestamptz;
    deleted        bigint;
BEGIN
    SET max_parallel_workers_per_gather = 0;
    SET enable_seqscan = off;

    SELECT MIN(created_at)
    INTO left_ts
    FROM return_requests
    WHERE created_at < end_timestamp;

    IF left_ts IS NULL THEN
        RAISE NOTICE 'No rows to delete';
        RETURN;
    END IF;

    LOOP
        EXIT WHEN left_ts >= end_timestamp;

        DELETE
        FROM return_requests
        WHERE created_at >= left_ts
          AND created_at < left_ts + batch_interval
          AND created_at < end_timestamp;

        GET DIAGNOSTICS deleted = ROW_COUNT;

        RAISE NOTICE 'Deleted % rows for time range (% .. %)',
            deleted,
            left_ts,
            left_ts + batch_interval;

        COMMIT AND CHAIN;

        left_ts := left_ts + batch_interval;
    END LOOP;
END
$$;

max_parallel_workers_per_gather = 0 limits parallel execution and helps keep the cleanup workload more predictable.

enable_seqscan = off discourages sequential scans and makes PostgreSQL prefer the index on the timestamp column. Use it carefully in production: if PostgreSQL still prefers a sequential scan, the batch may simply not be selective enough.

The interval should be small enough to make each batch selective. Ideally, PostgreSQL should use an Index Scan instead of scanning a large part of the table with a Bitmap Index Scan.

Delete on return_requests  (cost=0.44..790.25 rows=0 width=0)
  ->  Index Scan using return_requests_created_at_idx on return_requests  (cost=0.44..790.25 rows=735 width=6)
        Index Cond: ((created_at >= (now() - '20 days'::interval))
                AND (created_at < ((now() - '20 days'::interval) + '01:00:00'::interval))
                AND (created_at < (now() - '10 days'::interval)))

Fixed-size batching by integer

This is a more intuitive batching strategy: instead of splitting the cleanup by time ranges, we delete a fixed number of rows per iteration.

DO $$
DECLARE
    rows_deleted INTEGER;
    total_deleted BIGINT := 0;
    batch_size    INT := 1000;
    end_timestamp timestamptz := now() - interval '10 days';
BEGIN
    SET max_parallel_workers_per_gather = 0;
    SET enable_seqscan = off;

    LOOP
        WITH to_delete AS (
            SELECT id
            FROM return_requests
            WHERE created_at < end_timestamp
            LIMIT batch_size
        )
        DELETE FROM return_requests rr
        USING to_delete td
        WHERE rr.id = td.id;

        GET DIAGNOSTICS rows_deleted = ROW_COUNT;

        total_deleted := total_deleted + rows_deleted;

        RAISE NOTICE 'Deleted % rows in this batch (total: %)',
            rows_deleted,
            total_deleted;

        COMMIT AND CHAIN;

        EXIT WHEN rows_deleted < batch_size;

    END LOOP;
END $$;

The query uses the created_at index to find rows, but then performs an additional lookup by primary key for each selected row. This makes the query easy to control and predictable in terms of batch size, but slightly less efficient than deleting directly by a time range. The query could scan the whole dataset, but unmodified rows are not locked and not causing waits on application side.

Delete on return_requests rr  (cost=0.85..103.25 rows=0 width=0)
  ->  Nested Loop  (cost=0.85..103.25 rows=27 width=38)
        ->  Subquery Scan on td  (cost=0.43..31.97 rows=27 width=40)
              ->  Limit  (cost=0.43..31.70 rows=27 width=8)
                    ->  Index Scan using return_requests_created_at_idx on return_requests  (cost=0.43..31.70 rows=27 width=8)
                          Index Cond: (created_at < (now() - '10 days'::interval))
        ->  Index Scan using return_requests_pkey on return_requests rr  (cost=0.42..2.64 rows=1 width=14)
              Index Cond: (id = td.id)

Index does not exist

Creating an index

If you have enough disk space and can tolerate additional write pressure, creating an index may be the simplest solution.

Use CONCURRENTLY to build the index with less impact and without blocking normal reads and writes. Keep in mind that even this mode is still incompatible with some operations and takes longer than a regular index build.

Cleanup using a unique auxiliary column

This approach is a bit more subtle. Instead of batching by time, we scan the table using a unique index and apply the date condition as a filter. This lets PostgreSQL avoid an expensive sequential scan, even though created_at is not indexed.

Delete on promo_tokens  (cost=0.42..23.03 rows=0 width=0)
  ->  Index Scan using promo_tokens_pkey on promo_tokens  (cost=0.42..23.03 rows=1 width=6)
        Index Cond: ((id >= 100012) AND (id < 101012))
        Filter: (created_at < (now() - '10 days'::interval))

This is especially useful when there is no index on created_at, but there is a primary key or another indexed integer column.

DO
$$
DECLARE
    batch_size    bigint := 1000;
    end_timestamp timestamptz := now() - interval '10 days';
    left_id       bigint;
    right_id      bigint;
    deleted       bigint;
BEGIN
    SET max_parallel_workers_per_gather = 0;
    SET enable_seqscan = off;

    SELECT MIN(id), MAX(id)
    INTO left_id, right_id
    FROM promo_tokens;

    IF left_id IS NULL THEN
        RAISE NOTICE 'No rows to delete';
        RETURN;
    END IF;

    LOOP
        EXIT WHEN left_id > right_id;

        DELETE
        FROM promo_tokens
        WHERE created_at < end_timestamp
          AND id >= left_id
          AND id < left_id + batch_size;

        GET DIAGNOSTICS deleted = ROW_COUNT;

        RAISE NOTICE 'Deleted % rows for id range [% .. %)',
            deleted,
            left_id,
            left_id + batch_size;

        COMMIT AND CHAIN;

        left_id := left_id + batch_size;
    END LOOP;
END
$$;

Unlike time-based batching, the batch size here is fixed. This makes the workload more predictable and easier to control.

The main downside is that batches can be “wasted”: a range of ids may contain no rows matching the delete condition. PostgreSQL still has to scan the index and fetch heap tuples, only to discard them during filtering. This results in unnecessary CPU work and buffer activity without making actual progress.

Cleanup using a non-unique auxiliary column

Assume the worker_id column in the task_processing_events table has high selectivity, but is not unique. If the distribution of values is uneven and some worker ids have a large number of rows, using this column as a predicate can be expensive: a single batch may include hundreds of thousands or even millions of rows in the worst case.

A practical solution is to use the implicit system column ctid, which represents the physical location of a row on disk. It allows us to identify and delete specific rows without relying on a logical unique identifier.

DO $$
DECLARE
    rows_deleted  integer;
    total_deleted bigint := 0;
    batch_size    int := 1000;
    v_worker_id   int;
    v_worker_ids  int[];
    end_timestamp timestamptz := now() - interval '10 days';
BEGIN
    SET max_parallel_workers_per_gather = 0;
    SET enable_seqscan = off;

    SELECT array_agg(DISTINCT worker_id)
    INTO v_worker_ids
    FROM task_processing_events
    WHERE created_at < end_timestamp;

    IF v_worker_ids IS NULL THEN
        RAISE NOTICE 'No rows to delete';
        RETURN;
    END IF;

    FOREACH v_worker_id IN ARRAY v_worker_ids LOOP
        LOOP
            WITH to_delete AS (
                SELECT ctid
                FROM task_processing_events
                WHERE worker_id = v_worker_id
                  AND created_at < end_timestamp
                LIMIT batch_size
            )
            DELETE FROM task_processing_events tpe
            USING to_delete td
            WHERE tpe.ctid = td.ctid;

            GET DIAGNOSTICS rows_deleted = ROW_COUNT;

            total_deleted := total_deleted + rows_deleted;

            RAISE NOTICE 'worker_id=%, deleted=% rows in this batch, total=%',
                v_worker_id,
                rows_deleted,
                total_deleted;

            COMMIT AND CHAIN;

            EXIT WHEN rows_deleted < batch_size;
        END LOOP;
    END LOOP;
END $$;

Important to point out: ctid is not a persistent identifier. It can change after UPDATE, VACUUM FULL, CLUSTER, or table rewrites. It is safe as a temporary identifier within a single statement, not as something to store and reuse later.

Delete on task_processing_events tpe  (cost=0.42..331.15 rows=0 width=0)
  ->  Nested Loop  (cost=0.42..331.15 rows=1 width=36)
        ->  Subquery Scan on td  (cost=0.42..330.03 rows=1 width=36)
              ->  Limit  (cost=0.42..330.02 rows=1 width=6)
                    ->  Index Scan using task_processing_events_worker_id_idx on task_processing_events  (cost=0.42..330.02 rows=1 width=6)
                          Index Cond: (worker_id = 978)
                          Filter: (created_at < (now() - '10 days'::interval))
        ->  Tid Scan on task_processing_events tpe  (cost=0.00..1.11 rows=1 width=6)
              TID Cond: (ctid = td.ctid)

This approach is similar to batching via a unique key, but uses ctid as a temporary physical identifier. It allows efficient row-by-row deletion even when no suitable unique column exists. However, it relies on physical row locations and should be used carefully.

Deleting data based on a query

If the rows to delete are identified by a complex query with several joins, it may be better to materialize the result first.

Instead of recalculating the same join on every batch, we store the target row ids in a helper table and then use that table to drive the batch delete.

In this example, we delete rows from delivery_events, but the filtering condition is based on orders.created_at.

CREATE TABLE delivery_event_ids_to_delete AS
SELECT de.id
FROM delivery_events de
JOIN orders o ON o.id = de.order_id
WHERE o.created_at < now() - interval '10 days';

CREATE INDEX CONCURRENTLY delivery_event_ids_to_delete_id_idx
ON delivery_event_ids_to_delete (id);

DO $$
DECLARE
    rows_deleted  integer;
    total_deleted bigint := 0;
    batch_size    int := 10000;
BEGIN
    SET max_parallel_workers_per_gather = 0;
    SET enable_seqscan = off;

    LOOP
        WITH batch AS (
            DELETE FROM delivery_event_ids_to_delete
            WHERE id IN (
                SELECT id
                FROM delivery_event_ids_to_delete
                ORDER BY id
                LIMIT batch_size
            )
            RETURNING id
        )
        DELETE FROM delivery_events de
        USING batch b
        WHERE de.id = b.id;

        GET DIAGNOSTICS rows_deleted = ROW_COUNT;

        total_deleted := total_deleted + rows_deleted;

        RAISE NOTICE 'Deleted % rows in this batch (total: %)',
            rows_deleted,
            total_deleted;

        COMMIT AND CHAIN;

        EXIT WHEN rows_deleted < batch_size;
    END LOOP;
END $$;

DROP TABLE delivery_event_ids_to_delete;

This approach is useful when the selection logic is expensive. The complex join is executed once, and every batch works with a simple list of ids.

The helper table also preserves progress. If the cleanup is interrupted, you can continue from the remaining rows in delivery_event_ids_to_delete.

The trade-off is extra disk usage: you temporarily store ids and an index for them.

Batch update

Batch updates follow the same idea as batch deletes: instead of modifying all matching rows in one long transaction, we update a limited number of rows per iteration.

In this example, some return requests were incorrectly marked as processed. We reset them back to new so the application can process them again.

Each psql call runs one batch in its own transaction. The updated row ids are written to a file, which gives us a simple audit trail and helps understand how much progress was made.

#!/bin/bash
set -euo pipefail

DB_NAME="batch_delete_demo"
DB_USER="andrey.samsonov"
DB_HOST="localhost"
DB_PORT="5432"

OUTPUT_FILE="updated_ids.txt"
TEMP_FILE="batch_tmp.txt"

MAX_RUNS=1000
BATCH_SIZE=1000
CUTOFF_TIMESTAMP="2026-04-25 17:44:20.079898+03"

rm -f "$OUTPUT_FILE" "$TEMP_FILE"

for ((i=1; i<=MAX_RUNS; i++)); do
    psql "host=${DB_HOST} \
          port=${DB_PORT} \
          dbname=${DB_NAME} \
          user=${DB_USER} \
          target_session_attrs=read-write" \
        -v ON_ERROR_STOP=1 \
        -X \
        -q \
        -t \
        -A \
        -c "
WITH to_update AS MATERIALIZED (
    SELECT id
    FROM return_requests
    WHERE created_at > '${CUTOFF_TIMESTAMP}'::timestamptz
      AND status = 'processed'
    LIMIT ${BATCH_SIZE}
)
UPDATE return_requests rr
SET status = 'new'
FROM to_update tu
WHERE rr.id = tu.id
RETURNING rr.id;
" > "$TEMP_FILE"

    rows_updated=$(grep -c '[^[:space:]]' "$TEMP_FILE" || true)

    if [ "$rows_updated" -eq 0 ]; then
        echo "No more rows found. Exiting at iteration $i."
        rm -f "$TEMP_FILE"
        break
    fi

    cat "$TEMP_FILE" >> "$OUTPUT_FILE"

    echo "Iteration $i: updated $rows_updated rows."
done

rm -f "$TEMP_FILE"

echo "Done. Updated ids were written to $OUTPUT_FILE"

The important part is not the shell script itself, but the transaction boundary: every iteration updates only a limited set of rows and commits immediately. This keeps locks short-lived and makes the operation restartable.

This pattern is useful for cron jobs, application maintenance workers, and one-time operational fixes where restartability, logging, and external control are more important than minimizing connection overhead.

MySQL

Other database systems may require slightly different ways to organize batch operations. For example, in MySQL, this logic can be wrapped into a stored procedure.

DELIMITER $$

CREATE PROCEDURE batch_update_return_requests()
BEGIN
    DECLARE v_min_id BIGINT;
    DECLARE v_max_id BIGINT;
    DECLARE v_batch_size INT DEFAULT 1000;
    DECLARE v_cutoff DATETIME DEFAULT NOW() - INTERVAL 10 DAY;

    SELECT MIN(id), MAX(id)
    INTO v_min_id, v_max_id
    FROM return_requests
    WHERE created_at < v_cutoff
      AND status = 'processed';

    IF v_min_id IS NOT NULL THEN

        WHILE v_min_id <= v_max_id DO

            UPDATE return_requests
            SET status = 'new'
            WHERE created_at < v_cutoff
              AND status = 'processed'
              AND id BETWEEN v_min_id AND v_min_id + v_batch_size - 1;

            COMMIT;

            SET v_min_id = v_min_id + v_batch_size;

        END WHILE;

    END IF;
END$$

DELIMITER ;

CALL batch_update_return_requests();

This is the same integer-based batching pattern: the primary key defines the batch range, while the actual business condition is applied as an additional filter.

The procedure first finds the minimum and maximum id for rows that may need to be updated. Then it walks through that range in fixed-size chunks.

Each iteration updates only one id range, which keeps transactions smaller and reduces the risk of long locks, large rollback, and heavy replication lag.

One important difference from PostgreSQL examples is that the batching loop is implemented as a stored procedure. MySQL does not have PostgreSQL-style anonymous DO $$ ... $$ blocks for this kind of procedural logic.

Also note that with autocommit = 1, each UPDATE is committed automatically. The explicit COMMIT is mostly useful when autocommit is disabled or when you want to make the batch boundary obvious.

The main downside is the same as with integer-based batching in PostgreSQL: some id ranges may contain no matching rows, so some batches may do no useful work. This approach works best when id roughly correlates with the data being updated or deleted.

Summary

Batching is not about making DELETE or UPDATE magically cheap. The same rows still have to be changed, WAL still has to be generated, and dirty pages still have to be flushed eventually.

The real benefit is control: shorter locks, smaller transactions, preserved progress and smoother I/O.

If the operation is small, isolated, or strictly atomic by design, a single transaction may be fine. But once it starts touching a large amount of production data, batching is usually the safer operational choice.

If cleanup is a recurring operation, not a one-time fix, partitioning is usually the better option. You can simply drop old partitions, reclaiming space without dealing with bloat from large deletes.
